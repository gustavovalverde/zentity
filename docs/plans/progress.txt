--- Task 39: MCP HTTP Transport Auth (findings #1, #2, #3) ---
Status: COMPLETE

Changes:
- apps/mcp/src/auth/token-auth.ts: JWKS URL /api/auth/jwks → /api/auth/oauth2/jwks
- apps/mcp/src/transports/http.ts: removed unused AUTH_TOKEN_RE, cleaned up auth context (no fake DPoP key)
- apps/mcp/src/auth/api-client.ts: fixed pre-existing lint (async→sync serviceTokenFetch, let→const dpopNonce)
- apps/web/src/lib/trpc/server.ts: added resolveServiceTokenSession (timing-safe compare X-Zentity-Internal-Token, lookup user by X-Zentity-User-Id)

Architectural Decision:
DPoP relay (PRD choice) is architecturally impossible — DPoP proofs are URL-bound (htu claim),
so caller's proof for POST /mcp can't be reused for GET /api/trpc/*. Token's cnf.jkt is bound
to caller's key, so MCP can't mint new proofs. Implemented service token approach instead —
MCP validates caller via DPoP, then authenticates to web app as trusted internal service with
X-Zentity-Internal-Token + X-Zentity-User-Id. Web app uses timing-safe comparison and
buildSessionFromUserId. This is the standard microservice gateway pattern.

serviceTokenFetch kept (was going to be deleted per PRD) because it's correct; the bug was
the web app not recognizing the headers, now fixed.

Feedback loops: typecheck ✓, tests ✓ (109 MCP, 88 web unit), lint ✓ (changed files clean)

--- Task 41: Demo-rp Auth & Session Hardening (findings #4, #5, #6) ---
Status: COMPLETE

Changes:
- apps/demo-rp/src/lib/auth.ts: fetchUserInfo uses DPoP proof when dpopClients Map has entry
  for the access token. getToken stores DPoP client by access token after exchange.
  Module-level dpopClients Map bridges getToken → getUserInfo across genericOAuth callback chain.
- apps/demo-rp/src/lib/dpop.ts: exported DpopClient interface
- apps/demo-rp/src/app/api/ciba/route.ts: fetchTokenWithDpop returns dpop client. Token exchange
  and CIBA token callers store DPoP client in dpopByToken Map. Userinfo handler retrieves and uses
  stored DPoP client for proof-of-possession. Falls back to Bearer if no DPoP client stored.
- apps/demo-rp/src/app/api/auth/backchannel-logout/route.ts: replaced hardcoded JWKS URL + issuer
  with OIDC Discovery fetch (cached at module level). Added audience check using all registered
  client IDs. Removed ZENTITY_JWKS_URL env var dependency for BCL.

Architectural Decisions:
- DPoP client bridging via module-level Map keyed by access token. Necessary because genericOAuth's
  getToken and getUserInfo are separate callbacks — no way to thread state through the plugin.
  Map entries are consumed (deleted) on use, so no memory leak.
- CIBA route uses same pattern: dpopByToken Map bridges token exchange → userinfo across separate
  HTTP requests from the frontend.
- BCL audience check uses ALL registered client IDs (jose supports array). Token must match at least one.

Feedback loops: typecheck ✓, lint ✓
