## First-Party Apps — Phase 1: Authorization Challenge Endpoint
- Created POST /api/oauth2/authorize-challenge with 3-round OPAQUE flow
- Round 1: identifier + OAuth params → auth_session + challenge_type + server_public_key
- Round 2: auth_session + opaque_login_request → opaque_login_response
- Round 3: auth_session + opaque_finish_request → authorization_code
- DPoP key bound at round 1 and enforced on subsequent rounds
- Auth code stored in verification table (same SHA-256 format as oauth-provider)
- redirect_to_web returned for users without OPAQUE credentials
- authorization_challenge_endpoint added to discovery metadata
- Files: apps/web/src/app/api/oauth2/authorize-challenge/route.ts (new)
- Files: apps/web/src/lib/auth/well-known-utils.ts (discovery)
- Integration tests: 7 tests covering all 3 rounds, error cases, session validation
- Test infra: real OPAQUE server setup in vitest.setup.mts, authChallengeSessions in resetDatabase
- Token exchange: auth code exchangeable at /oauth2/token without redirect_uri
- Token response includes auth_session for future step-up re-authentication
- Vendor tarball patched: redirect_uri optional, session creation deferred, auth_session pass-through
- Auth code hash fixed: base64url encoding to match oauth-provider's defaultHasher
- Phase 1 COMPLETE: all acceptance criteria met

## First-Party Apps — Phase 2: redirect_to_web + Credential Resolution
- Credential resolution: OPAQUE > EIP-712 wallet > redirect_to_web (passkey-only)
- Unknown users get OPAQUE challenge (timing-safe, indistinguishable from real OPAQUE)
- redirect_to_web includes PAR request_uri when code_challenge present (FPA draft §3.2)
- PAR record stored in haipPushedRequests, consumable at standard authorize endpoint
- No request_uri when no code_challenge (per spec)
- code_challenge/method now optional in schema + request validation
- Tests: passkey-only → redirect_to_web + request_uri, no PKCE → no request_uri, unknown → opaque
- Phase 2 COMPLETE

## First-Party Apps — Phase 5: First-Party Enforcement + Hardening
- `firstParty` boolean column added to oauthClients schema (default: false)
- Non-first-party clients receive `redirect_to_web` unconditionally (no credential resolution)
- Rate limiting: 10 requests/minute per IP via sliding window (X-Forwarded-For)
- Lazy session cleanup: expired authChallengeSessions deleted on each request
- DPoP key continuity enforced across all 3 rounds (already existed, now tested)
- Tests: non-first-party → redirect_to_web, rate limit → 429, per-IP isolation, DPoP key switch → invalid_session
- Phase 5 COMPLETE

## First-Party Apps — Phase 3: EIP-712 Wallet Challenge Flow
- Extracted shared EIP-712 utils (buildDefaultTypedData, verifyEip712Signature, nonceIdentifier) into plugins/eip712/utils.ts
- Plugin server.ts refactored to use shared utils (no behavior change)
- 2-round wallet auth: Round 1 returns nonce + typed_data, Round 2 verifies signature → authorization_code
- Identifier supports both email and Ethereum address (0x-prefixed, 42 chars)
- Wallet context (address, chainId, nonce) stored in opaqueServerState as JSON
- Nonce stored in verifications table (same as plugin), consumed on round 2
- verifyEip712Signature catches invalid signature errors (returns false vs throwing)
- issueAuthorizationCode extracted as shared helper (used by OPAQUE and EIP-712 finish)
- Tests: eip712 challenge by email, by address, full 2-round flow, token exchange, invalid signature
- Phase 3 COMPLETE

## First-Party Apps — Phase 4: Step-Up Re-Authentication
- Token endpoint returns HTTP 403 + auth_session for FPA clients when CIBA acr_values unsatisfied
- Non-FPA clients continue to receive 400 interaction_required (unchanged behavior)
- enforceCibaTokenAcr detects firstParty flag, creates challenge session with DPoP JKT + scope + acrValues
- Challenge endpoint accepts step-up re-entry: auth_session (no challengeType) → resolves credentials → starts OPAQUE/EIP-712
- resolveCredentialType helper extracted (shared credential resolution logic)
- Step-up sessions gated by !session.challengeType (prevents hijacking regular sessions)
- Optional PKCE (code_challenge/code_challenge_method) accepted on step-up re-entry for public clients
- DPoP key consistency enforced from CIBA token request through challenge rounds
- acrValues column added to authChallengeSessions schema
- Tests: FPA→403+auth_session, non-FPA→400, step-up re-entry→OPAQUE, full CIBA→challenge→tokens, DPoP consistency
- Phase 4 COMPLETE
