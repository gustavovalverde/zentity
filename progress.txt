# Architectural decisions
- Concurrent release test accepts 400 OR 410 for loser — Node.js single-threaded async means the second caller may re-read DB after status="claiming", hitting status guard (400) instead of atomic claim race (410). Both prevent double PII delivery.

## OIDC Assurance-Aware ID Tokens — Phase 2: at_hash (Access Token Hash Binding)
- Pure function: computeAtHash in src/lib/assurance/oidc-claims.ts
- Vendor tarball patch: sequenced access token creation before ID token in createUserTokens, passed accessToken to customIdTokenClaims via createIdToken
- Type def patch: added accessToken?: string to customIdTokenClaims info param
- Wired in auth.ts: computes at_hash from accessToken + client signing alg (metadata.id_token_signed_response_alg, default RS256)
- Discovery: at_hash added to claims_supported
- 7 unit tests for computeAtHash (RS256, ES256, EdDSA, ML-DSA-65, unknown alg, same-alg equivalence, different-token divergence)
- 1 integration test verifying at_hash matches recomputed SHA-256 left-half of access_token

# Architectural decisions
- Vendor sequencing: broke Promise.all parallelism in createUserTokens — access token created first (await), then refresh token + ID token in parallel. Enables at_hash in customIdTokenClaims without post-hoc re-signing.
- Signing alg resolution: reads metadata.id_token_signed_response_alg from client metadata (same source of truth as jwt-signer.ts), defaults to RS256.
