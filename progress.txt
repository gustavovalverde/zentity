## Task 06: CIBA Boundary Evaluation Pipeline
Status: Complete

### Changes
1. Fixed authorization_details type flow: `evaluateBoundaries` now accepts `AuthorizationDetail[]` instead of `string | null`
2. Added `normalizeAuthorizationDetails(raw: unknown)` — accepts parsed array, JSON string, or null
3. `tryAutoApprove` uses normalizeAuthorizationDetails at the boundary instead of string-only guard
4. Changed match semantics: boundaries return three-state `{ match, pass, reason }` or `{ match: false }`
5. Purchase/custom boundaries: no matching detail → `match: false` (neutral) instead of `pass: true` (vacuous approval)

- Parse once at the boundary (`normalizeAuthorizationDetails`), not inside evaluation
- Three-state result eliminates vacuous pass: boundaries that don't apply are skipped
- Scope boundaries always match (every CIBA request has scopes), so a user with only a scope boundary still works
