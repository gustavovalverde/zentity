# Tasks completed in this session

## 1. Push subscription schema + env vars + VAPID setup (PRD §1, §env)
- Created `push_subscriptions` table with userId FK, unique endpoint, p256dh/auth keys
- Added VAPID env vars (server + client) to env.ts, .env.example
- Setup script generates VAPID keys via `web-push generate-vapid-keys --json`
- Installed `web-push` + `@types/web-push`

## 2. Server-side push sender (PRD §2)
- `sendWebPush()` in src/lib/push/web-push.ts
- Dispatches to all user devices, auto-cleans stale 410 subs
- Fire-and-forget: errors logged, never thrown

## 3. CIBA notification integration (PRD §6)
- sendNotification fires push + email in parallel via Promise.allSettled
- Push short-circuits silently when VAPID not configured

## 4. Push subscription API routes (PRD §3)
- POST /api/push/subscribe — upserts browser push subscription
- POST /api/push/unsubscribe — removes subscription scoped to user

## 5. Service worker + client manager (PRD §4-5)
- push-sw.js: Approve/Deny action buttons, requireInteraction, tag dedup
- client.ts: subscribeToPush(), unsubscribeFromPush(), getPushState()

## 6. CIBA listing banner (PRD §8)
- PushNotificationBanner component on /dashboard/ciba
- Shows enable/disable/denied states based on Notification.permission

## 7. PWA manifest (PRD §7)
- src/app/manifest.ts serving /manifest.webmanifest
- Enables installability and iOS Web Push

## 8. Unit tests
- 5 tests for sendWebPush: all devices, 410 cleanup, no subs, no throw, no VAPID

## 9. Integration tests for push API routes (PRD §Testing)
- 7 tests: subscribe 401/store/upsert/400, unsubscribe 401/delete/noop
- Added pushSubscriptions to resetDatabase() in db-test-utils

## 10. Fix: SW inline approve/deny field name (auth_req_id not authReqId)
- CIBA endpoints expect snake_case `auth_req_id` per RFC 9126
- SW was sending camelCase `authReqId` — would silently fail inline approve

## 10. Next.js PWA best practices alignment
- SW registration: added scope: '/', updateViaCache: 'none' per Next.js guide
- SW headers: no-cache, restrictive CSP, correct content-type in next.config.ts
- getPushState now async, checks actual PushManager subscription (not just permission)
- Banner distinguishes "subscribed" vs "unsubscribed" (permission granted but no sub)

# Decisions made and why
- Schema follows ciba.ts pattern (defaultId, timestamp_ms, FK cascade)
- VAPID keys optional — system gracefully degrades to email-only
- Both VAPID_PUBLIC_KEY and NEXT_PUBLIC_VAPID_PUBLIC_KEY hold same value
- push-sw.js is a separate file from coi-serviceworker.js (no conflict)
- Service worker uses fetch with credentials: 'same-origin' for inline approval
- Banner is a client component embedded in server component page
