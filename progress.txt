## Task 08 (Part 2): PII Routing to Userinfo via Claims Parameter
Status: Complete

### Changes
1. `customIdTokenClaims` now PEEKS ephemeral PII (non-destructive)
   - PII stays available for userinfo delivery
   - `claims.id_token` filter still applied
2. `customUserInfoClaims` now CONSUMES ephemeral PII (terminal consumer)
   - When `claims.userinfo` requests identity fields, PII delivered via userinfo
   - Backward compat: no `claims` param → PII only in id_token
3. Added `peekEphemeralClaimsByUser` to ephemeral-identity-claims.ts
4. Tests: 3 new (peek vs consume, cross-client safety)

### Architectural decision
- id_token hook peeks, userinfo hook consumes — userinfo is terminal consumer
- If RP never calls userinfo, 5-min TTL cleans up ephemeral PII
- Cross-client safety guard preserved in peek (returns null for 2+ entries)

## Task 06: Custodial FROST Email-Triggered Signing
Status: Complete (web-side wiring; signer binary unchanged)

### Changes
1. `signRecoveryChallenge` accepts `endpointOverrides` param
   - Overlays custom signer endpoints by participant index
   - Used when CUSTODIAL_SIGNER_URL is configured
2. `approveGuardianProcedure` detects custodialEmail guardians
   - Passes CUSTODIAL_SIGNER_URL as endpoint override for the custodial guardian's participant index
   - Custodial signer receives the same guardian_assertion JWT as all other signers

- endpointOverrides Map injected at the challenge level, not at frost-service global config

- CUSTODIAL_SIGNER_ID remains for signer identity correlation in logging/admin UI (future use)
