## Phase 1.1: Enable CIBA ping mode server-side (Plan §Phase 1, AC 1-5)
- Enabled `deliveryModes: ["poll", "ping"]` in ciba() plugin config

# Key decisions made and reasoning (architectural decisions)
- `resolveClientNotificationEndpoint` queries `oauthClients.metadata` directly via Drizzle (already imported in auth.ts) rather than going through the better-auth adapter — simpler, no indirection
- Ping mode stores notification endpoint/token per CIBA request (not per client) — allows per-request endpoint override via `client_notification_uri` in bc-authorize body

## Phase 3.2: PII encryption helpers (Plan §Phase 3, AC 2)
- `sealApprovalPii` — generates 256-bit release handle, encrypts PII with AES-GCM, returns handle (base64url), hash (hex), ciphertext + IV (base64)
- `unsealApprovalPii` — decrypts PII using the release handle
- `hashReleaseHandle` — SHA-256 of handle for DB lookup
- 10 unit tests covering round-trip, uniqueness, hash matching, wrong key/corrupted ciphertext rejection

## Phase 3.3: PII release endpoint (Plan §Phase 3, AC 3-9)

- Status lifecycle: approved → redeemed (410 on replay), expired (410 with auto-transition)
- PII filtering by approved scopes via `filterIdentityByScopes`
- 8 integration tests: full redemption, one-time use, expired, missing handle, unknown handle, no bearer, authorization_details, scope filtering

- 13 integration tests covering all acceptance criteria:

  - Access→Access with audience binding via `resource` parameter
